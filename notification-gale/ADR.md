# notification-gale 設計判断記録（ADR）

このファイルは、本ゲーム（未来からの同意要求｜SAFARI FUTURE RELAY）の設計判断とその経緯を記録するものです。
**改修（特にAIエージェントによる改修）の前に必ず読んでください。** ここに書かれた判断は「なぜ今そうなっているか」の正であり、経緯を知らずに変更すると過去に潰したバグや調整済みの体験バランスを壊します。仕様の全体像は `README.md`、判断の経緯はこの ADR が担当します。

**追記ルール:**
- 新しい設計判断をしたら、このファイルの末尾に `ADR-nnn`（連番）で追記する。
- 既存の判断を覆す場合は、新しい ADR を追記した上で、旧 ADR のステータスを `Superseded by ADR-nnn` に変更する（旧 ADR 本文は消さない）。
- 各 ADR には関連コミットハッシュを残す。

## ADR-001: 呼気検知の閾値は strongRatio=2.2 とし、URLクエリで上書き可能にする

- 日付: 2026-07-13（初期設計は 2026-07-11）
- ステータス: Accepted
- 文脈: コアギミック「強い息で広告を吹き飛ばす」は、環境音ベースラインに対する相対RMS比（`DETECT.strongRatio`）で「強い息」を判定する。初期値 3.0 は誤発火（話し声・環境音での発動）を防ぐには安全だったが、実機では正しく息を吹きかけても検知が渋く、ギミックに気づいたのに反応しない＝理不尽な詰まりを生んでいた。
- 決定: `game.js` の `DETECT.strongRatio` デフォルトを 3.0 → 2.2 に緩和（コミット 39afc93）。話し声・定常環境音を弾くゲート（ノイズ様判定＝平坦度/ゼロ交差率、持続時間 `shold`=120ms、クリップ除外、2.5秒ごとのベースライン再キャリブレーション）は不変。また、判定パラメータはすべて URL クエリで上書き可能（例 `?debug=1&strong=2.4&shold=100`。`strong` / `shold` / `ratio` / `flat` / `zcr` / `clip` / `span`）。
- 結果・注意:
  - 閾値をいじるときはコードを書き換える前に、まず `?debug=1&strong=...` で実機検証してから確定値をコードに反映する（このデバッグ経路を消さないこと）。
  - 「感度を上げる＝strongRatioを下げる」方向の変更は誤発火リスクとトレードオフ。話し声ゲート（ノイズ様判定）を同時に緩めないこと。
  - Playwright 検証は fake audio（`--use-fake-device-for-media-stream` 等）＋ `window.gameDebug.setBaseline(0.004)` で決定的に行う（README「デバッグ」節）。
  - 関連コミット: 4d30f9e（初期実装）, 39afc93（3.0→2.2）

## ADR-002: マイク使用中インジケータは CSS で常時非表示（ヒント強度 > プライバシー告知の常時表示）

- 日付: 2026-07-12
- ステータス: Accepted
- 文脈: プライバシー配慮として「マイク回線 使用中／音声は録音・送信されません」チップ＋レベルメーターを常時表示していたが、プレイ中に「マイクが動いている」ことが常に見えると「息を吹きかける」という答えのヒントになりすぎることが判明した（本作は操作をUIで教えない方針＝ADR-006 が核）。
- 決定: `styles.css` の `.mic-indicator` に `display: none !important` を当てて常時非表示にする（コミット 3da42c8）。**JS側のロジック（`elements.micIndicator` の hidden/クラス切替、`updateMicIndicator`）は意図的に残してある**——CSSで確実に隠しつつ、将来表示に戻す/デバッグ表示する余地を保つため。プライバシーへの配慮は冒頭のマイク許可ゲートの文言と、音声を端末外へ送信しない実装（Web Audio 特徴量のみ）で担保する。
- 結果・注意:
  - `game.js` に micIndicator 操作コードが残っているのは死にコードではなく意図的。「JSで参照しているのにCSSで消えている＝バグ」と誤診して `display:none` を外さないこと（外すとヒント過多に戻る）。
  - `styles.css` の当該箇所（`.mic-indicator` 定義直上）にも同旨のコメントがある。消さないこと。
  - 関連コミット: 3da42c8

## ADR-003: 演出系フルスクリーン画面は「safe center ＋ overflow-y:auto」で可変コンテンツに耐えさせる

- 日付: 2026-07-13
- ステータス: Accepted
- 文脈: クリア画面 `#clear-screen` は固定演出前提の `justify-content: center`＋overflow:visible だったが、後から「広告を購入していた場合に未来の自分に叱責される演出（購入一覧＋合計請求額）」ブロックを条件付きで追加したところ、コンテンツが画面高を超えると下部に到達できずスクロールも不能になった（実質的な進行不能バグ）。
- 決定: `.title-screen, .clear-screen` に `justify-content: safe center` ＋ `overflow-y: auto` ＋ `-webkit-overflow-scrolling: touch` ＋ `overscroll-behavior: contain` を設定（コミット 39afc93）。safe center は「収まるときは中央寄せ、あふれるときは上寄せ」でフレックスの上端クリップを回避する。
- 結果・注意:
  - タイトル/クリア等のフルスクリーン演出画面に要素を追加するときは、この overflow 設定を外さないこと。`justify-content: center`（safe なし）へ戻すと同じバグが再発する。
  - 今後、条件分岐でコンテンツ量が変わる画面を新設する場合も同パターンを最初から適用すること。
  - 関連コミット: 3da42c8（叱責ブロック追加＝バグの発生源）, 39afc93（修正）

## ADR-004: キャッシュバスターは index.html の `?v=` 3箇所を手動で揃えて更新する

- 日付: 2026-07-13（運用確立）
- ステータス: Accepted
- 文脈: 静的HTML＋GitHub Pages 配信のためビルド工程がなく、ブラウザ（特にXアプリ内ブラウザ）が古い JS/CSS をキャッシュして「直したはずのバグが直っていない」事故が起きる。
- 決定: `index.html` が読み込む `styles.css` / `analytics.js` / `game.js` に `?v=<日付+連番>`（例 `?v=20260713a`）を付け、**これら3ファイルのいずれかを変更したら、index.html 内の3か所の `?v=` を同じ新しい値へ手で更新する**（コミット b83dcbb が運用例。同日2回目は `b` に進める）。
- 結果・注意:
  - CSS/JS を変更したのに `?v=` を更新し忘れると、本番で変更が反映されない。**改修コミットとセットで必ず更新すること**（3箇所の値が揃っているかも確認）。
  - 現状は手動運用。自動化（pre-commit hook 等）は未導入（導入する場合は新ADRとして追記）。
  - 関連コミット: b83dcbb

## ADR-005: 現在の正は「2010年風Web × 敵対的広告 × 呼気吹き飛ばし」構成（初期実装からの全面リビルド）

- 日付: 2026-07-11
- ステータス: Accepted
- 文脈: 初期実装（コミット 4d30f9e, game.js 588行）は公開直後に体験フローの見直しが必要と判断され、同日中にダイアログ演出を軸とした構成へ全面リビルドされた（コミット c4389e6 "rebuild notification puzzle around misrouted dialogs"。game.js/styles.css の過半を書き換え）。その後 3da42c8 で広告詳細・購入・叱責演出が加わり現在の形になった。
- 決定: 現在の正の構成は README.md 記載のとおり——本体UIは「2010年頃の古めかしいWeb」（未来人による不完全な再現、という世界律で恣意パラメータを物語化）、広告は「現代の胡散臭いWeb広告の再現」（中身だけ未来技術）、体験フローは タイトル→マイク許可ゲート→ダウンロード→緊急警告→広告氾濫→検知→管理人連絡（絶望）→呼気で吹き飛ばし→承認、の**同一画面内進行（別画面へ遷移しない）**。
- 結果・注意:
  - 初期実装（4d30f9e時点）のコード・演出を「元に戻す」参照先にしないこと。正は現HEAD＋README。
  - 「本体UIが古い／広告が現代風」の時代差は手抜きではなく世界設定（README「設計意図」節）。トンマナ統一の名目で揃えないこと。
  - リビルドの詳細な動機はコミットメッセージ以上の記録がないため、「初期構成では違和感→誤解→体験→回収の流れが弱かった」というのは推定。
  - 関連コミット: 4d30f9e, c4389e6, 3da42c8, f13dd2e（タイトル文言に未来側の事情を補足）

## ADR-006: 「吹き飛ばす」操作は UI で教えない。誘導は環境ヒント＋長時間停滞時の段階後出しのみ

- 日付: 2026-07-11
- ステータス: Accepted
- 文脈: コアギミック（息で広告を吹き飛ばす）をボタンやチュートリアルで教えると謎ではなくなる。「操作指示・答えをUIに直接書かない。ヒントは詰まった後に段階的後出し」という制作方針（nazo-hint-design）に従う。
- 決定: 操作の告知を廃し、環境に遠回しの手掛かりだけを散らす——(1) ステータスバーの `build FUKITOBASU`（index.html 43行目付近）、(2) 検知フェーズと管理人応答の「回線は大きな風速に弱い状態」という文言、(3) 氾濫広告に1個だけ紛れたクリック可能な「未署名の受信」（2段階の遠回しな示唆、読まなくてもクリア可）。救済は氾濫到達から約90秒/約150秒の停滞時のみ段階ヒントを後出しし、一度でも吹き飛ばしに成功（first_blow）したら以降出さない。マイク拒否・非対応時は手動払い（長押しスワイプ/Delete）へフォールバック。
- 結果・注意:
  - 「わかりにくいから」とマイクボタン・操作説明・チュートリアルを追加しないこと（ADR-002 のインジケータ非表示も同じ方針の一部）。難易度調整はヒントの後出しタイミング・文言・閾値（ADR-001）で行う。
  - `FUKITOBASU`・「風速に弱い」の文言・ヒント受信は難易度装置なので、コピー変更時も「風・吹く」のモチーフを壊さないこと。
  - 進行保証（連絡ボタン常時可押、ヒント受信の z-index は全広告より上＝物理・手動払いの対象外）を崩さないこと。
  - 関連コミット: 4d30f9e, c4389e6

## ADR-007: モバイル可視域は `--vvh`（visualViewport追従）を正とし、固定UIは safe-area と被り回避の配置制約を守る

- 日付: 2026-07-11（初期実装＋同日の被り修正）
- ステータス: Accepted
- 文脈: Xアプリ内ブラウザ等では `100vh` が実際の可視域とずれ、下部UIが隠れる。また画面の四隅に置いた固定UI（戻るスタンプ、復帰導線）が他要素やアニメーションと被る問題が公開直後に2件発生した（95621e4: ready時のスタンプアニメが transform で動いて位置が不安定→box-shadow のみのアニメへ変更、216b79c: デスクトップで戻るスタンプが被る→右配置から左配置 `left:42px` へ）。
- 決定: `game.js` で `visualViewport` を監視して CSS 変数 `--vvh` を更新し（コード中に「Xアプリ内ブラウザ対策」と明記、1592行目付近）、高さに応じて `html.vvh-short`（≤640px）/`html.vvh-tiny`（≤520px）クラスを付与、`--vvh` 変化時は広告物理の基準（`recomputeAdBases`）も測り直す。レイアウトは `var(--vvh, 100dvh)` と `env(safe-area-inset-*)` を全面的に使用。固定UIの配置は被り修正後の値（`.return-button` はデスクトップで左配置等）が正。
- 結果・注意:
  - 高さ指定に生の `100vh` を使わないこと（`var(--vvh, 100dvh)` を使う）。ステータスバー/コンソール等の位置計算は `env(safe-area-inset-*)` 込みの現行値を保つ。
  - 位置が意味を持つ固定UIに transform 系アニメーションを付けない（95621e4 の再発防止。動かすなら box-shadow 等レイアウトに影響しないプロパティで）。
  - `vvh-short` / `vvh-tiny` の分岐を消すと縦600px級（アプリ内ブラウザ）で崩れる。ヒント受信の可視位置も vvh-tiny を考慮済み（game.js 689行目付近）。
  - 関連コミット: 95621e4, 216b79c, 4d30f9e
